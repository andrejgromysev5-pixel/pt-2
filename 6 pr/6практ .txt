ПР №6. Анализ сетевого трафика: tcpdump и Wireshark
1. Разбор строки tcpdump
Строка: 12:34:56.789012 IP 127.0.0.1.44321 > 127.0.0.1.8080: Flags [S], seq 123, length 0
| Поле            | Значение        | Что означает                                                               |
| --------------- | --------------- | -------------------------------------------------------------------------- |
| 12:34:56.789012 | Время           | Метка времени — часы:минуты:секунды.микросекунды                           |
| IP              | Протокол        | Указывает, что это IP-пакет                                                |
| 127.0.0.1.44321 | Источник        | IP-адрес и порт отправителя (127.0.0.1 — loopback, порт 44321 — эфемерный) |
| 127.0.0.1.8080  | Назначение      | IP-адрес и порт получателя (127.0.0.1, порт 8080 — сервер)                 |
| Flags \[S]      | Флаги TCP       | Флаг SYN — запрос на установление соединения (первый пакет TCP handshake)  |
| seq 123         | Sequence number | Начальный порядковый номер для отслеживания пакетов                        |
| length 0        | Длина payload   | Размер данных в пакете — 0 байт, т.к. SYN-пакет не содержит данных         |

2. Перехваченные данные HTTP
Что видно в трафике:
В HTTP-трафике видно всё содержимое запросов и ответов в открытом тексте: заголовки, тело запроса, параметры формы. В частности, при отправке формы с логином и паролем строка username=admin&password=supersecret123 видна полностью в ASCII-представлении пакета (tcpdump -A).
Кто в реальной сети может так же прочитать этот трафик:
Атакующий в той же локальной сети (через ARP-spoofing, поддельный DHCP, порт-мирроринг на коммутаторе)
Системный администратор с доступом к сетевому оборудованию
Провайдер интернета (если трафик идёт через него без VPN)
Владелец публичной Wi-Fi точки
Злоумышленник, внедривший сниффер на скомпрометированный хост
3. Follow TCP Stream
Что видит злоумышленник, перехвативший этот трафик:
При использовании функции Follow TCP Stream в Wireshark виден полный диалог клиент-сервер:
Красным цветом — всё, что отправил клиент: HTTP-заголовки (Host, User-Agent, Content-Type), тело запроса с параметрами формы (username=admin&password=supersecret123), заголовок Authorization с Base64
Синим цветом — ответ сервера: статус-код, заголовки ответа, HTML-страница с ошибкой или содержимым
Скомпрометированные данные:
Учётные данные пользователя (логин и пароль в открытом виде)
Содержимое всех HTTP-запросов и ответов
Сессионные cookie (если передаются)
Заголовок Authorization (Base64 — легко декодируется)
Полная структура взаимодействия клиента с сервером

| Что наблюдаем             | HTTP (порт 8080)       | HTTPS (порт 443)                                         |
| ------------------------- | ---------------------- | -------------------------------------------------------- |
| IP-адреса сторон          | Видны                  | Видны (IP не шифруется)                                  |
| Порт назначения           | Виден (8080)           | Виден (443)                                              |
| Заголовки HTTP            | Видны полностью        | Не видны — зашифрованы TLS                               |
| Содержимое запроса (body) | Видно открытым текстом | Не видно — зашифровано                                   |
| Логин и пароль            | Видны напрямую         | Невозможно определить                                    |
| Имя домена (SNI)          | Видно в заголовке Host | Видно в SNI (Server Name Indication) TLS — до шифрования |


Вывод: HTTPS шифрует содержимое коммуникации, но метаданные (IP, порт, домен через SNI) остаются видимыми.
5. DNS и приватность
Какие домены видны в DNS даже при HTTPS:
При захвате DNS-запросов (порт 53) видны все домены, к которым обращается клиент:
example.com
github.com
google.com
Что это означает для приватности:
HTTPS шифрует содержимое страницы, но не шифрует DNS-запросы. Это создаёт канал утечки метаданных:
Провайдер или сниффер видит, какие сайты посещает пользователь
Видно время и частоту посещений
Возможно профилирование интересов пользователя по DNS-запросам
Решение: использовать DNS-over-HTTPS (DoH) или DNS-over-TLS (DoT), которые шифруют DNS-запросы.
6. Base64
Декодированная строка:
dmFzeWE6TXlQYXNzd29yZDIwMjQ= → vasya:MyPassword2024
Что такое Base64:
Base64 — это метод кодирования бинарных данных в текстовый формат с использованием 64 символов (A-Z, a-z, 0-9, +, /). Это не шифрование, а только представление данных в виде текста, безопасного для передачи по текстовым протоколам.
Почему Base64 в HTTP Basic Auth не является защитой:
Base64 не использует ключ — декодирование тривиально (команда base64 -d)
Любой, кто перехватил трафик, может мгновенно получить логин и пароль
Это эквивалентно "спрятать" письмо в прозрачный конверт — форма другая, но содержимое видно
HTTP Basic Auth передаёт Base64 в открытом виде внутри HTTP-заголовка, который сам по себе не шифруется
Что нужно использовать вместо: HTTPS (TLS) для шифрования всего канала, либо более современные методы аутентификации (Bearer tokens, OAuth 2.0, клиентские сертификаты).
7. tshark vs tcpdump vs Wireshark

| Инструмент    | Когда использовать                                                                                        |
| ------------- | --------------------------------------------------------------------------------------------------------- |
| **tcpdump**   | Быстрый захват в консоли, серверы без GUI, автоматизация в скриптах, минимальные ресурсы                  |
| **tshark**    | Консольный анализ pcap-файлов, статистика по протоколам, фильтрация на сервере, интеграция в CI/CD        |
| **Wireshark** | Глубокий графический анализ, Follow TCP Stream, визуальная фильтрация, обучение, расследования инцидентов |
8. Каналы утечки через сеть
Меры защиты, закрывающие сетевой канал утечки:
HTTPS/TLS — шифрование всего HTTP-трафика
VPN — шифрование всего трафика в туннеле
DNS-over-HTTPS (DoH) / DNS-over-TLS (DoT) — шифрование DNS-запросов
HSTS (HTTP Strict Transport Security) — принудительное использование HTTPS
Сетевое сегментирование — VLAN, firewall, ограничение доступа
Порт-мирроринг только для авторизованных систем — контроль за снифферами
Обнаружение ARP-spoofing — защита от атак man-in-the-middle в LAN
Использование современных протоколов — SSH вместо Telnet, SFTP вместо FTP
Выводы
В ходе работы было продемонстрировано, что HTTP-трафик полностью передаётся открытым текстом, и любой участник сети (или атакующий с доступом к сети) может перехватить и прочитать конфиденциальные данные: логины, пароли, токены, содержимое запросов. HTTPS решает проблему шифрованием содержимого, но метаданные (DNS-запросы, IP-адреса, SNI) остаются видимыми. Base64 не является методом защиты — это только кодирование, легко обратимое. Для полноценной защиты сетевого канала необходим комплексный подход: HTTPS + DoH/DoT + VPN + сетевое сегментирование.
Ответы на контрольные вопросы
78. Что такое promiscuous mode и зачем он нужен снифферу?
Promiscuous mode (неразборчивый режим) — это режим работы сетевой карты, при котором она принимает все пакеты, проходящие через сеть, а не только адресованные ей. В обычном режиме сетевая карта отбрасывает пакеты, не предназначенные для её MAC-адреса. Promiscuous mode нужен снифферу для перехвата чужого трафика в сегменте сети (например, в хаб-среде или при ARP-spoofing). В современных коммутируемых сетях promiscuous mode на одном хосте не даёт доступа к чужому трафику без дополнительных атак (port mirroring, ARP-spoofing).
79. Чем отличается capture filter от display filter в Wireshark?
Capture filter (фильтр захвата) — применяется на уровне libpcap/tcpdump до записи пакета. Работает на уровне BPF (Berkeley Packet Filter), фильтрует по низкоуровневым параметрам (IP, порт, протокол). Не видит содержимого пакета (например, нельзя отфильтровать "пакеты со словом password"). Пример: port 8080, host 192.168.1.1
Display filter (фильтр отображения) — применяется после захвата, на уровне Wireshark/tshark. Работает с уже декодированными протоколами, может фильтровать по любым полям любого протокола. Пример: http.request.method == "POST", frame contains "password"
80. Что такое Follow TCP Stream и для чего используется?
Follow TCP Stream — функция Wireshark, которая собирает все пакеты одного TCP-соединения в хронологическом порядке и отображает их как единый поток данных. Используется для:
Восстановления полного диалога клиент-сервера
Просмотра содержимого HTTP-запросов и ответов в едином тексте
Анализа передаваемых файлов и данных
Обнаружения скрытой информации, разбитой на несколько пакетов
Визуального разделения трафика: клиент → сервер (красным), сервер → клиент (синим)
81. Почему Base64 в HTTP Basic Auth не является защитой? Что нужно использовать вместо?
Base64 — это кодирование, а не шифрование. Оно не использует ключ, не обеспечивает конфиденциальность, любой может декодировать за миллисекунды. HTTP Basic Auth передаёт Base64 внутри HTTP-заголовка, а HTTP сам по себе не шифруется. Это эквивалентно передаче пароля открытым текстом с незначительным кодированием.
Вместо этого нужно использовать:
HTTPS (TLS) для шифрования всего канала
Более современные схемы аутентификации: Bearer tokens (JWT), OAuth 2.0, OpenID Connect
Клиентские сертификаты TLS
Digest authentication (хотя он тоже устарел)
82. Какие данные утекают даже при использовании HTTPS? Как это исправить?
Даже при HTTPS утекают метаданные:
DNS-запросы — видны домены, которые посещает пользователь
IP-адреса — видны серверы, к которым идёт подключение
SNI (Server Name Indication) — в TLS 1.2 и ниже имя домена передаётся в открытом виде при установлении соединения
Размер пакетов и тайминг — можно анализировать паттерны трафика
Размер TLS-сертификата — может указывать на конкретный сайт
Исправление:
DNS-over-HTTPS (DoH) или DNS-over-TLS (DoT) — шифрование DNS
ESNI / ECH (Encrypted Client Hello) в TLS 1.3 — скрытие имени сервера
VPN — шифрование всего трафика включая метаданные
Tor — многослойное шифрование и анонимизация
83. Сисадмин говорит: «у нас свитч, а не хаб — значит сниффинг невозможен». Он прав?
Нет, не прав. Коммутатор (switch) действительно изолирует трафик на уровне MAC-адресов и не пересылает чужой unicast-трафик на все порты, как это делал хаб. Однако сниффинг всё равно возможен через:
ARP-spoofing (ARP poisoning) — подделка ARP-ответов для перенаправления трафика через атакующего
Port mirroring (SPAN) — злоумышленник с административным доступом к свитчу может настроить зеркалирование порта
MAC-flooding — переполнение CAM-таблицы свитча, заставляющее его работать как хаб
DHCP-spoofing — подмена DHCP-сервера для перенаправления трафика
** Rogue access point** — поддельная точка доступа с контролируемым трафиком
