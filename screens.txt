1. Пользователи и группы
Созданные пользователи и их роли:
Пользователь	Группа	Роль
alice	developers	Администратор проекта
bob	developers	Разработчик
carol	auditors	Аудитор
Поля /etc/passwd (разбор строки alice)
Пример вывода `getent passwd alice`:
```
alice:x:1001:1001::/home/alice:/bin/bash
```
№ поля	Значение в примере	Описание
1	`alice`	Имя учётной записи (login name)
2	`x`	Пароль — символ `x` означает, что хэш хранится в /etc/shadow
3	`1001`	UID — числовой идентификатор пользователя
4	`1001`	GID — числовой идентификатор основной группы
5	(пусто)	GECOS — необязательное описание (полное имя, контакты)
6	`/home/alice`	Домашняя директория пользователя
7	`/bin/bash`	Командная оболочка (shell), запускаемая при входе
```
Поля: имя пользователя; хэш пароля (алгоритм SHA-512, `$6$`); дата последней смены пароля (дни с 01.01.1970); минимальный срок; максимальный срок; предупреждение за N дней; период неактивности; дата истечения.
Почему недоступен обычным пользователям: файл имеет права `-rw-r-----` (640) и владельца `root:shadow`. Это сделано намеренно — если хэши паролей будут доступны всем, злоумышленник сможет провести атаку по словарю или перебором офлайн, не оставляя следов в системных журналах. В /etc/passwd пароль был доступен исторически, поэтому механизм теневых паролей (shadow passwords) и был введён как мера защиты.

2. Права доступа chmod/chown
Итоговые права `/srv/project/code`: `chmod 770`, владелец `alice:developers`
Файл/каталог	Права (символьно)	Права (числово)	Владелец	Группа
/srv/project/code	drwxrwx---	770	alice	developers
/srv/project/reports	drwxr-xr-x	755	root	root
/srv/project/logs	drwxr-xr-x	755	root	root
reports/q1.txt	-rw-r-----	640	alice	auditors


Результаты проверок
bob (в группе developers) при правах 750:
bob не смог создать файл командой `touch /srv/project/code/test.py`.
Причина: при правах 750 группа имеет только `r-x` (чтение + выполнение, без записи). Право `w` (запись) у группы отсутствует, а именно оно нужно для создания файлов внутри каталога. Чтобы разрешить запись — нужно сменить на `chmod 770`.
bob при правах 770:
bob смог создать файл `hello.py`. Теперь группа имеет `rwx` (7), запись разрешена.
carol (в группе auditors, не в developers) при правах 770:
carol не смогла войти в `/srv/project/code`.
Вывод: `Permission denied`
Причина: разряд "остальные" (other) = `0` — никаких прав. Carol не является ни владельцем (alice), ни членом группы developers, поэтому попадает в категорию `other` и доступ полностью закрыт

Файл q1.txt — разбор прав
```
-rw-r----- alice auditors reports/q1.txt
```
Разряд	Символьно	Числово	Кто	Права
Владелец	rw-	6	alice	Чтение + запись
Группа	r--	4	auditors	Только чтение
Остальные	---	0	все остальные	Нет доступа
Может ли carol прочитать q1.txt? Да — carol входит в группу `auditors`, которой дано право чтения (`r--`). Команда `cat /srv/project/reports/q1.txt` выполнится успешно.

```
Чем ACL лучше стандартной модели в данной ситуации:
ACL позволяет назначить права конкретному пользователю (carol) без изменения группы файла и без добавления carol в группу developers. Это точечное управление доступом реализует принцип наименьших привилегий: carol получает ровно столько прав, сколько нужно для аудита, не больше.

4. sudo-политики
Пользователь	Разрешено	Ограничение
alice	Всё (NOPASSWD)	Нет
bob	/usr/bin/apt, /usr/bin/apt-get	Только пакетный менеджер
carol	/usr/bin/journalctl, /bin/cat /var/log/*	Только просмотр логов
Почему alice использует NOPASSWD, а у bob и carol ограниченный доступ:
Реализован принцип наименьших привилегий (Principle of Least Privilege) — каждый пользователь получает ровно те права, которые необходимы для выполнения его задач:
alice — администратор, ей нужен полный контроль над системой, постоянный запрос пароля снизил бы оперативность работы.
bob — разработчик, ему достаточно устанавливать пакеты; доступ к journalctl или смене прав файлов ему не нужен и потенциально опасен.
carol — аудитор, её задача только читать логи, она вообще не должна изменять систему.
Такая конфигурация соответствует группе мер УПД (Управление правами доступа) Приказа ФСТЭК №17.

5. PAM
/etc/pam.d/sudo
```
auth       include      common-auth
account    include      common-account
session    required     pam\_permit.so
session    include      common-session-noninteractive
```
/etc/pam.d/login (фрагмент)
```
auth       required     pam\_securetty.so
auth       requisite    pam\_nologin.so
auth       include      common-auth
...
```
Модуль pam_unix.so — основной модуль Unix-аутентификации. Проверяет пароль пользователя по хэшу из `/etc/shadow` с использованием алгоритма SHA-512. Выполняет всю стандартную проверку учётных данных Linux.
Что означает `required`:
Ключевое слово `required` означает, что модуль обязателен для успешной аутентификации — если он вернёт ошибку (пароль неверный), аутентификация в итоге завершится неудачей. Однако PAM продолжит обрабатывать следующие модули в стеке (не прерывает цепочку немедленно) — это сделано для противодействия атакам по времени, чтобы нельзя было определить, на каком именно модуле произошёл отказ.
Другие управляющие флаги: `requisite` (немедленно прервать при ошибке), `sufficient` (достаточно для успеха, если предыдущие не провалились), `optional` (результат не влияет на итог).

Контрольные вопросы

44. chmod 640 — что это значит и кто может читать: 640 = владелец — чтение и запись (rw-), группа — только чтение (r--), остальные — нет доступа (---). Читать файл могут владелец и члены группы владельца. 

45. setuid-бит и sudo: setuid — бит на исполняемом файле, который запускает программу всегда от имени владельца файла, автоматически, без проверки того, кто запускает программу. Пример: /usr/bin/passwd имеет setuid root — любой пользователь может изменить свой пароль, хотя /etc/shadow принадлежит пользователю root. sudo — явный запрос на предоставление привилегий с проверкой политики, ведением журнала и (обычно) вводом пароля. Sudo более контролируемый: можно ограничить доступ к определенным командам. 

46. Почему хэши хранятся в /etc/shadow, а не в /etc/passwd: /etc/passwd должен быть доступен для чтения всем пользователям, иначе не будут работать базовые команды (ls, ps и т. д.). Если бы хэши хранились там, любой пользователь мог бы скачать их и подобрать пароли методом перебора. /etc/shadow доступен только пользователю root, что исключает такую угрозу.

47. Почему опасно использовать sudo bash: Боб получит интерактивную оболочку с правами root без каких-либо ограничений. Из bash можно выполнить абсолютно любую команду — фактически это полный root-доступ. Любое ограничение в файле sudoers теряет смысл. 

48. su и sudo с точки зрения безопасности:

49 Запретить пользователю sudo выполнять определенные действия, не удаляя его из группы: Добавьте в /etc/sudoers через visudo: bob ALL=(ALL) !ALL Или создайте файл /etc/sudoers.d/bob: sudo visudo -f /etc/sudoers.d/bob Добавьте строку: bob ALL=(ALL) !ALL Это явно запрещает пользователю bob выполнять все команды через sudo, даже если он входит в группу sudo.
